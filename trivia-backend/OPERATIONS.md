# Trivia Backend Operations

## What This Is

The backend handles data processing for the trivia directory: scraping provider sites, importing and deduplicating venues and events, enriching venues via the Google Places API, and generating recurring event occurrences on a weekly cron. All data lives in Supabase (Postgres + PostGIS).

## Setup

You'll need Node.js 18+ and pnpm.

```bash
cd trivia-backend
pnpm install
cp .env.example .env
```

Fill in `.env`:

```bash
SUPABASE_URL=https://<project>.supabase.co
VITE_SUPABASE_URL=https://<project>.supabase.co
VITE_SUPABASE_ANON_KEY=<anon key>
SUPABASE_ANON_KEY=<anon key>
SUPABASE_SERVICE_ROLE_KEY=<service role key>
GOOGLE_PLACES_API_KEY=<places key>
```

The service role key is required for the scheduler and batch imports (it bypasses RLS). Keep it out of the frontend.

If you're bootstrapping a fresh database, apply the files under `../supabase/migrations/` (repo-root `supabase/migrations/`) via the Supabase CLI (`supabase db reset`) or the Supabase SQL Editor.

## Data Model

- `venues` — location with PostGIS point, Google Places metadata, status
- `events` — recurring event templates (venue + day_of_week + start_time + provider)
- `event_occurrences` — concrete dated instances generated by the scheduler
- `trivia_providers` — Challenge Entertainment, NerdyTalk, BrainBlast, etc.
- `admin_users` / `provider_users` — auth linkage (admin vs provider-scoped)
- `api_usage_log` — Google Places call tracking for rate limiting
- `system_logs` / `notifications` — scheduler audit trail

A unique constraint on `events(venue_id, event_type, day_of_week, start_time, provider_id) WHERE is_active = true` prevents duplicate templates.

Venue `status` and event `status` are `pending | approved | rejected`. Event occurrences have `status` of `scheduled | confirmed | cancelled | completed`; public app shows `scheduled` and `confirmed`.

## Data Ingestion Pipeline

Scrape → import → validate → photo processing.

```bash
# 1. Scrape a provider
pnpm scrape-nerdytalk-city-aware
pnpm scrape-brainblast

# 2. Import scraped JSON into Supabase (dedupes, inserts venues + events)
pnpm import-nerdytalk
pnpm import-brainblast
pnpm import-venues providers/challenge-entertainment/ce-data-<date>.json

# 3. Validate new venues against Google Places (addresses, lat/lng, photos)
pnpm validate-places validate --limit 100

# 4. Process venue photos (downloads, resizes, uploads to Supabase storage)
pnpm process-photos
```

Scraped data lands in `providers/<provider>/` as JSON. Import scripts upsert venues, apply manual corrections (e.g. `nerdytalk-venue-corrections.json`), and insert event templates with deduplication via `utils/event-deduplication.js`.

Each provider lives in its own directory with its scraper, importer, and data files. See `providers/README.md` for the layout convention.

## Google Places Validation

The Places API is capped at 1000 calls/day with a 110ms delay between calls. Daily usage is tracked in `api_usage_log`.

```bash
pnpm validate-places stats                       # show pending/verified/failed counts
pnpm validate-places validate --limit 100        # validate unverified venues
pnpm validate-places validate --dry-run --limit 10
pnpm validate-places validate --failed --limit 50
pnpm validate-places validate --needs-review --limit 20
```

For annual re-validation, run `--limit 900` periodically; the script prioritizes venues with the oldest `last_places_api_call`.

Verification statuses: `pending`, `verified`, `failed`, `needs_review`.

## Scheduled Jobs

### Sunday-night scheduler

`sunday-night-scheduler.js` runs every Sunday at 22:00 server time via cron. It:

1. Generates `event_occurrences` for the next 4 weeks from every active weekly `event`, skipping dates that already have an occurrence and respecting `start_date`/`end_date`.
2. Writes an admin summary (events needing confirmation in the next 7 days) to logs and `notifications`.
3. Deletes unconfirmed occurrences older than 2 weeks; keeps confirmed ones for history.
4. Logs to `system_logs` and the scheduler log file.

Only `frequency = 'weekly'` events are processed today. `once`, `monthly`, `biweekly` are not implemented.

Manual commands:

```bash
pnpm scheduler:dry-run    # safe preview, no writes
pnpm scheduler            # real run
pnpm generate-events      # equivalent job entry point (jobs/generate-weekly-events.js)
pnpm monitor              # health check
pnpm monitor:json         # JSON report for scripting
```

### Email policy

No emails go to providers or venue owners. Admin summaries are logged (and optionally emailed to `jtdman+tfadmin@gmail.com` if SMTP is wired up in `jobs/generate-weekly-events.js`). Providers are expected to manage events through the admin UI.

## Adding a New Provider

1. Create `providers/<provider-name>/` with a scraper (`scrape-*.js`) that writes a JSON file of venues + events. Include city context in every venue record so Google Places matches correctly — missing cities caused the Casa Mexicana and ML Rose ambiguity issues with NerdyTalk.
2. Create an importer (`import-<provider>.ts`) that upserts a row into `trivia_providers`, dedupes venues, and inserts event templates using `utils/event-deduplication.js`.
3. Add pnpm scripts for scrape and import to `package.json`.
4. Optionally add a `<provider>-venue-corrections.json` for known bad matches.
5. Run the pipeline end-to-end against a small limit, then validate with Google Places.

See `providers/nerdytalk/` for a complete example.

## Common Operational Tasks

### Regenerate this week's occurrences manually

```bash
pnpm scheduler:dry-run   # verify what it'll do
pnpm scheduler           # run it
```

### Check event counts

```sql
SELECT COUNT(*) FROM event_occurrences WHERE occurrence_date >= CURRENT_DATE;
SELECT COUNT(*) FROM event_occurrences
  WHERE status = 'confirmed' AND occurrence_date >= CURRENT_DATE;
```

### Check for duplicate event templates (should be 0)

```sql
SELECT COUNT(*) - COUNT(DISTINCT CONCAT(venue_id, '|', event_type, '|', day_of_week, '|', start_time)) AS duplicates
FROM events WHERE is_active = true;
```

### Clean up existing duplicates (if the constraint catches any)

```bash
pnpm fix-duplicates:dry-run
pnpm fix-duplicates
```

### Check venue photo status

```sql
SELECT COUNT(*) FROM venues WHERE photo_url IS NULL AND status = 'approved';
```

Then run `pnpm process-photos` to backfill.

### Add an admin or link a provider to a user

Create the user in Supabase Auth > Users, copy the UUID, then:

```sql
INSERT INTO admin_users (user_id, email, role)
VALUES ('<uuid>', '<email>', 'super_admin');

-- or link a provider account
INSERT INTO provider_users (user_id, provider_id, role)
VALUES ('<user uuid>', '<provider uuid>', 'admin');

UPDATE trivia_providers
SET auth_user_id = '<user uuid>', auth_status = 'approved'
WHERE id = '<provider uuid>';
```

## Logs and Monitoring

- Scheduler log (local dev): `trivia-backend/scheduler.log`
- Scheduler log (prod): `/var/log/trivia-scheduler.log` (or `/var/log/trivia-events.log` depending on which cron entry installed it)
- Database audit trail: `system_logs` table
- API call tracking: `api_usage_log` table
- Notifications queue: `notifications` table

Tail logs:

```bash
tail -f scheduler.log
grep -i error scheduler.log
```

Health check:

```bash
pnpm monitor              # human-readable status
pnpm monitor:json | jq .  # machine-readable
pnpm monitor:continuous   # keep running
```

Monitor looks for: scheduler run freshness, database connectivity, recent occurrence counts, and error rates in `system_logs`.

## Troubleshooting

**No occurrences generated this week.** Check `events.is_active = true`, `frequency = 'weekly'`, valid `day_of_week`, and that `start_date <= today <= end_date`. Run `pnpm scheduler:dry-run` to see what it would create.

**Google Places quota hit.** Check `api_usage_log` for today's count. Wait until UTC midnight or raise the daily quota in your Google Cloud project.

**Duplicate events appear in the UI.** The unique constraint should prevent new ones; for old data run `pnpm fix-duplicates`.

**Venue matched to wrong city.** Check the scraper captured the city header correctly and re-run with corrections in `<provider>-venue-corrections.json`. Then re-validate that venue: `pnpm validate-places validate --failed --limit 5`.
